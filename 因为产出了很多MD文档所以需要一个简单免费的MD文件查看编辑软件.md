### 1.2 关于"很多"的定量分析

"很多"是多少？

这是一个看似简单但实际上极其深刻的问题。因为"很多"是一个相对概念。对于一个刚接触编程的大一新生来说，5 个 `.md` 文件就是很多了。对于一个工作了三年的后端工程师来说，可能要 50 个才算多。对于一个写了十年技术博客的人来说，500 个可能都不算什么。对于 Linus Torvalds 来说——算了，Linus 应该不怎么写 Markdown，他忙着在邮件列表里骂人。

但我们可以尝试给"很多"一个量化的定义。我提出以下标准，我称之为 **"MD 文件焦虑指数"（Markdown File Anxiety Index, MFAI）**：

- **MFAI = 0**：你有 0 个 `.md` 文件。你是谁？你怎么进来的？你确定你需要这个软件吗？
- **MFAI = 1**：你有 1-10 个 `.md` 文件。你很健康。你的文件管理习惯良好。你不需要任何特殊的软件。关闭这篇文档，去过你的正常生活吧。

- **MFAI = 3**：你有 51-200 个 `.md` 文件。你的 Finder 在打开某些文件夹时会卡顿零点几秒。这零点几秒不是因为文件太多，而是因为 Finder 在试图理解你的文件命名逻辑——它失败了。

## 第二章：关于"简单"这个需求

### 2.1 简单是什么

简单是什么？

你可能会觉得这个问题的答案很简单。但如果"简单"的定义本身是简单的，那么就不会有人把"简单"作为一个需求专门列出来了。你去餐厅吃饭不会跟服务员说"请给我一碗简单的米饭"——因为米饭默认就是简单的。你之所以在软件名字里专门写了"简单"，恰恰说明你已经被太多不简单的软件伤害过了。你是一个创伤后应激障碍患者，而你的创伤来源是——工具栏上的第 47 个按钮。

让我用一个类比来解释"简单"。

想象你站在一个自动售货机前面。你想买一瓶水。一个简单的自动售货机是这样的：你看到水，你投币，你按按钮，水掉出来。完事。

一个不简单的自动售货机是这样的：你走过去，它先让你扫码关注一个公众号。然后让你注册一个会员。然后问你要不要充值 100 元送 10 元。然后给你推荐了一款你不需要的能量饮料。然后问你是否接受用户协议。然后让你选择配送方式——等等什么？自动售货机还有配送方式？然后你终于找到了那瓶水，你点了购买，它告诉你"该商品暂时缺货，为您推荐以下替代品"，然后给你看了三瓶果汁和一包辣条。

你只是想买一瓶水。

你只是想看一个 `.md` 文件。

这两件事的本质是一样的。你有一个简单的需求，但这个世界执意要把它搞复杂。

### 2.2 关于"简单"的历史

在遥远的过去——比如说 2004 年——John Gruber 创造了 Markdown。他的初衷就是"简单"。用最简单的纯文本标记来表达格式。不需要学 HTML。不需要学 LaTeX。不需要学任何东西。你只需要知道 `#` 是标题，`*` 是斜体，`**` 是加粗。完了。这就是 Markdown 的全部。

但人类有一种与生俱来的本能——把简单的东西搞复杂。

于是我们有了 GitHub Flavored Markdown。然后是 CommonMark。然后是 MultiMarkdown。然后是 Markdown Extra。然后是 R Markdown。然后是 MDX。每一个变体都在原始的 Markdown 基础上加了一些东西。表格。脚注。任务列表。数学公式。Mermaid 图表。嵌入 React 组件。

到最后，Markdown 已经变得和它当初试图替代的那些东西一样复杂了。这就像一个人因为嫌汽车太复杂而发明了自行车，然后其他人在自行车上加了电动机、GPS、蓝牙音箱、手机支架、儿童座椅、后视镜、转向灯、防盗报警器、行车记录仪、和一个 USB 充电口——最后这辆自行车比汽车还重。

但这不是 Markdown 的错。Markdown 本身仍然是简单的。就像水本身仍然是简单的——即使有人在水里加了气泡、柠檬片、薄荷叶、和一种你不确定是不是化学合成的"天然果味"。

### 2.3 当代 Markdown 编辑器的七宗罪

让我们来数一数，那些不简单的 Markdown 编辑器都犯了哪些罪：

**第一宗罪：注册**

你只是想打开一个文件。软件跟你说："请先创建一个账户。"为什么？为什么你需要一个账户才能打开一个存在于你自己硬盘上的文件？这就像你想进自己家的卧室但门上贴了一张告示："请先在物业登记并提交身份证复印件。"这是你的家。这是你的文件。让我进去。

**第二宗罪：同步**

"您的文档已同步到云端！"我没有让你同步。我没有请求同步。我不需要同步。我的文件在我的硬盘上，我在我的电脑前，我可以直接打开它。你把它同步到云端之后发生了什么？我的网络用了一点流量。你的服务器多存了一份数据。NSA 可能多了一份"某程序员的购物清单"的副本。谁受益了？没有人。除了那个因此获得了更多服务器存储预算的产品经理。

**第三宗罪：模板**

"从模板开始！我们为您准备了 47 种精美模板！"我不需要模板。我要打开一个已经存在的文件。我已经写好了。内容已经在里面了。你给我一个模板干什么？你是不是觉得我写的东西不够好看？你是不是在用一种很委婉的方式告诉我我的文档排版很丑？

好吧，也许我的文档排版确实很丑。但那是我的丑。我的丑不需要你来管。

**第四宗罪：教程**

八个字。四个词。两个动词。两个名词。零个形容词——等等，"MD"算形容词吗？它修饰了"文件"，所以从语法上来说它在起形容词的作用。但"MD"本身是一个缩写，全称是 Markdown，而 Markdown 是一个专有名词，所以它在以名词的身份充当形容词。这在语言学上叫做名词形容词化使用（noun adjunct）。

### 4.2 关于"查看"

"查看"这个词，你以为你懂。其实你不懂。或者说你懂，但你没有意识到你懂的程度有多深。

当你"查看"一个 `.txt` 文件时，你看到的就是字。字是什么样子就是什么样子。没有悬念。没有惊喜。就像你看一面白墙——墙就是墙，白就是白，你不会期待墙上突然冒出一朵花来。

但当你"查看"一个 `.md` 文件时，情况就复杂了。你看到的不应该是原始的源码——那些 `#` 号、`**` 号、`-` 号、`` ` `` 号——你看到的应该是它们**代表**的东西。`#` 应该变成一个大标题。`**` 应该让文字变粗。`-` 应该变成一个圆点。`` ` `` 应该让代码有一个灰色的背景。

这意味着"查看"Markdown 文件是一个**翻译**行为。你不是在看文件本身，你是在看文件的**翻译结果**。就像你不是在看一本法语书，你是在看这本法语书的中文翻译。原始的法语还在那里，但你信任翻译者的能力，所以你只看翻译后的版本。

那么问题来了：如果翻译者翻译错了呢？如果你的 Markdown 渲染引擎把 `# 标题` 渲染成了普通文本呢？如果它把 `**加粗**` 显示成了 `**加粗**` 呢？如果它把你精心排版的表格变成了一坨缩成一团的文字呢？

你会愤怒。你会觉得被背叛了。你会关掉这个软件，去用另一个。但另一个可能会把你的 Mermaid 图表渲染成一段乱码。你再换一个，这个连 LaTeX 公式都不支持。你再换一个，这个支持所有功能但要收费。你再换——你已经回到了第三章。你被困在了一个循环里。

你知道吗，`cat` 命令永远不会翻译错。因为它根本不翻译。它给你看的就是原文。原汁原味。你看到的 `#` 就是 `#`，`*` 就是 `*`。没有歧义。没有误解。就像一个永远不会说谎的人——因为他根本不说话。

但你不想用 `cat`。因为你想看到漂亮的渲染效果。你想看到标题变大、代码变绿、引用块有一条优雅的竖线。你想要一个会说谎的人——不对，你想要一个会翻译的人。一个翻译得既准确又好看的人。这就是你对这个软件的全部要求。

翻译准确，翻译好看，翻译免费。

这个要求不过分吧？

过分。但你不在乎。因为你已经安装了这个软件。它已经在你的电脑里了。它翻译得怎么样？也许不错。也许有 bug。但它在那里。它是免费的。你和它之间已经建立了一种关系——一种基于零元购的、毫无负担的、随时可以卸载但你永远不会去卸载因为你懒的关系。

### 4.3 关于"编辑"

"编辑"。

让我先说一个故事。

从前有一个人，他打开了一个 `.md` 文件。他看到第三行有一个错别字。"的"写成了"得"。他想改过来。他把光标移到那个字上。他按了退格键。他打了一个"的"。他按了 Command+S。

但为了让这三秒钟成为可能，有人花了三个月写了一个文本编辑器。编辑器要处理光标定位、文本选择、复制粘贴、撤销重做、自动缩进、语法高亮、快捷键绑定、输入法兼容、大文件性能优化、滚动位置记忆……

你不需要知道这些。你只需要知道你能改字就行了。你打开文件，改一个字，保存，关闭。就像你用铅笔在纸上划掉一个字再写一个新的一样自然。

但铅笔不需要处理撤销重做。铅笔不需要兼容输入法。铅笔不需要三个月的开发时间。铅笔只需要一截木头和一根石墨芯。

### 4.4 查看 + 编辑 = 薛定谔的文档

Markdown 编辑器界有一场旷日持久的圣战：**左右分栏** vs **所见即所得**。

左右分栏派（又称"源码原教旨主义者"）认为：Markdown 的灵魂在于纯文本。你应该能看到每一个 `#`、每一个 `*`、每一个 `` ` ``。隐藏源码就是背叛 Markdown 的初衷。就像你用滤镜拍照就是背叛了真实——虽然你的脸确实在滤镜下好看了 47%，但那不是真实的你。

所见即所得派（又称"我不在乎源码长什么样给我看结果就行"派）认为：没有人想看源码。源码是给机器看的。人类应该看渲染后的效果。你写 `**加粗**` 的目的不是为了看到 `**加粗**`，而是为了看到**加粗**。手段和目的不要搞混。

这两派谁对？都对。谁错？也都错。因为这个问题的本质不是"哪种方式更好"，而是"你在什么时候需要哪种方式"。当你在写作的时候，你可能需要看源码，因为你要精确控制格式。当你在阅读的时候，你可能需要看渲染效果，因为你只关心内容。

所以最好的 Markdown 编辑器应该同时支持两种模式，并且可以一键切换——但这样一来，你又需要记住一个快捷键了。你的大脑里已经存了 47 个快捷键。Command+C，Command+V，Command+Z，Command+S，Command+Tab，Command+Space，Command+Shift+3，Command+Option+Esc（用来强制退出那些卡死的 App），Command+Shift+.（用来显示隐藏文件），还有那个你总是记不住的截屏快捷键——是 Command+Shift+4 还是 Command+Shift+5？你每次都要试两遍。

现在你还要记住一个"切换查看/编辑模式"的快捷键。你的大脑存储空间还够吗？

你看了一眼你的 241 个 `.md` 文件。你看了一眼你脑子里的 47 个快捷键。你叹了口气。

然后你继续阅读这篇废话文档。因为阅读废话是唯一不需要记住任何快捷键的事情。

如果你把这三个数字念出来，它听起来像一个摩斯密码。短-长-短。SOS？不，不是求救信号。虽然从某种意义上来说，任何 `0.x.x` 版本的软件都是在向用户发送一种微妙的求救信号："我还没做完，但我已经发布了，因为我需要一些正反馈来继续下去，否则这个项目会死在我的硬盘里，和其他 47 个未完成的 side project 一起。"

让我们用不必要的细致程度来解剖这三个数字。

**0——主版本号**

零。虚无。空。什么都没有。

主版本号是 0，在语义化版本规范（SemVer）里意味着：这个软件处于初始开发阶段。任何东西都可能改变。接口不稳定。功能不完整。用户体验不保证。换句话说——如果你用了这个软件然后它把你的文件吃了，你不能告它。

但你知道什么最有趣吗？互联网上有无数个主版本号为 0 的软件已经在生产环境中运行了五年以上。它们的开发者从来没有把版本号升到 1。因为一旦升到 1，就意味着你"正式发布"了，你就有了"责任"。主版本号 0 是一张免死金牌——"这个软件还在开发中"可以解释一切 bug、一切缺失功能、一切不合理的设计决策。

主版本号 0 是程序员最后的避风港。

**1——次版本号**

一。壹。the loneliest number。

次版本号是 1 意味着：从 0.0.x 到 0.1.0，发生了至少一次功能性的变更。这是一个里程碑。这意味着作者经历了以下心路历程：

- 0.0.1："哦，能编译了。"
- 0.0.2："哦，能运行了。"
- 0.0.3："哦，能打开文件了。"
- 0.0.4："哦，不会崩溃了。"（大部分时候。）
- 0.0.5 到 0.0.17：一系列无人关心的修复和调整，每个版本之间相隔 2 到 47 天不等，取决于作者是否处于"今天我要把这个项目做完"的短暂亢奋状态中。
- 0.1.0："行了。能用了。我要发布了。不能再拖了。再拖下去我就又要开一个新的 side project 了。"

你看，`0.1.0` 不仅仅是一个版本号。它是一段创业史。它浓缩了一个程序员从热情萌芽到濒临放弃再到强行发布的完整心理曲线。每一个 `0.1.0` 背后都站着一个在深夜里问自己"我为什么要做这个"的灵魂。

**0——修订号**

又一个零。

修订号是 0 意味着：自从 0.1.0 发布以来，没有进行过任何 bug 修复。

这有三种可能的解释：

1. 这个软件完美无缺，没有任何 bug——就像一个刚出生的婴儿没有任何债务一样，从技术上来说是正确的，但你知道这只是时间问题。
2. 这个软件有 bug，但还没有人用到——因为还没有人用这个软件。因为它昨天才发布。因为知道它的人只有作者本人，以及作者发了一条朋友圈但只有三个人点了赞而且其中一个是作者的妈妈。
3. 这个软件有 bug，有人用到了，但没有人报告——因为用户觉得"算了，免费的，要什么自行车"。

无论是哪种情况，修订号 0 都散发着一种新鲜出炉的气息。就像刚出锅的烧饼一样——烫手，但你还是想咬一口。

### 5.2 版本号的一生

如果这个软件没有在 0.1.0 就死掉（大多数 side project 都死在了这个阶段，死因通常是"作者对一个新的技术栈产生了兴趣"），那它未来的版本号可能会经历以下轨迹：

- **0.1.1**：修复了一个"打开超大文件时会卡死"的 bug。作者发现这个 bug 是因为他试图用自己的软件打开了这篇废话文档。
- **0.1.2**：修复了一个"中文标点符号渲染异常"的 bug。因为作者之前只用英文测试过。这就是为什么你永远不应该只用英文测试。
- **0.2.0**：加了暗色模式。这是所有软件的必经之路。就像所有人类都会经历青春期一样——不一定愉快，但不可避免。加暗色模式的原因有两个：第一，用户要求。第二，作者在凌晨两点写代码的时候被自己的白色界面闪瞎了眼。
- **0.3.0**：加了文件树侧边栏。因为你的 241 个文件不可能每次都手动找到路径然后拖进窗口。你需要一个侧边栏来展示它们。然后你会发现侧边栏里的文件太多了，你需要搜索。于是就有了——
- **0.4.0**：加了搜索功能。搜索功能的实现花了比预期多三倍的时间。因为"搜索"听起来很简单——不就是在文件里找字符串吗？但实际上你需要处理大小写敏感、正则表达式、全文搜索 vs 文件名搜索、搜索结果高亮、搜索结果排序……你最终发现你在实现一个小型搜索引擎。你开始理解为什么 Google 值那么多钱了。
- **0.5.0**：加了标签页。因为你需要同时打开七个文档来对比它们的内容。七个标签页。就像你浏览器里那 47 个永远不会关闭的标签页一样。
- **0.9.0**：在 0.5.0 到 0.9.0 之间发生了什么？没有人知道。可能加了一些功能。可能删了一些功能。可能重构了三次。可能换了两次 UI 框架。这段历史就像你高中时期的日记一样——它存在过，但你不想回顾。
- **1.0.0**：🎉 主版本号终于变成 1 了！发布公告写了 3000 字。配图精美。Product Hunt 上发了帖子。Hacker News 上也发了帖子。Reddit 上被人转发了。然后评论区有人说："这跟 Obsidian 有什么区别？"作者看到这条评论后沉默了十分钟，然后关掉了浏览器。
- **1.0.1**：发布后两小时紧急修复。因为 1.0.0 有一个在发布前的测试中没有被发现的 bug——文件保存后会多出一个空行。这就是为什么 1.0.0 从来不应该被信任。1.0.1 才是真正的 1.0。
- **2.0.0**：完全重写。名字改成了「因为产出了很多MD文档所以需要一个简单免费的而且有AI功能的而且支持云同步的而且可以协同编辑的MD文件查看编辑软件」。名字比 1.0 长了 40%。功能也多了 400%。但用户数只多了 4%。

当然，以上纯属虚构。这个软件大概率会停在 0.1.0 不动了。就像你床头柜上那本你两年前买的书一样——你一直说要读完的，但你永远停在了第三章。

## 第六章：关于 aarch64 架构

### 6.1 一个你从来没有想过、以后也不会想的问题

你有没有想过，当你双击一个 `.dmg` 文件的时候，你的电脑在做什么？

没有。你没有想过。就像你喝水的时候不会想"水分子是如何通过半透膜进入我的细胞的"一样。你只是喝了。它解渴了。完事。

但今天我要逼你想一下。因为这篇文档的目的就是让你思考那些你本来不需要思考的事情。

当你双击那个 `.dmg` 文件的时候，你的 Mac 会挂载一个虚拟磁盘，读取里面的 `.app` 包，而这个 `.app` 包里面有一个可执行文件——一个由 0 和 1 组成的二进制序列。这些 0 和 1 被送到你的处理器里，处理器根据它的**指令集架构**来理解这些 0 和 1 的含义。

你的处理器的指令集架构是 `aarch64`。

`aarch64`。让我们来拆解这个词：

- **a**：Apple？ARM？Awesome？都不是。这个"a"代表的是"AArch"——ARM Architecture 的缩写。是的，"aarch"本身就是一个缩写的缩写。这在技术界很常见。你的工作中也充满了缩写的缩写——比如 API 是 Application Programming Interface 的缩写，但你把"API"本身又缩写成了"接口"。所以当你说"这个接口有问题"的时候，你实际上是在说"这个 Application Programming Interface 有问题"，但你连"API"都懒得说，更不用说全称了。人类的语言在不断压缩。最终有一天我们可能会用一个字来表达一整句话。那个字可能是"嗯"。
- **arch**：Architecture，架构。注意不是"arch"那个 Linux 发行版。虽然如果你在用 Arch Linux，你大概率会在任何话题下想到 Arch Linux，就像你大概率会在对话中提到你在用 Arch Linux 一样——"顺便说一句，我用的是 Arch Linux。"
- **64**：64 位。意味着你的处理器一次可以处理 64 位的数据。64 位有多大？2 的 64 次方等于 18,446,744,073,709,551,616。这是一个十八万亿亿级别的数字。你的处理器可以寻址这么多字节的内存。你实际有多少内存？8GB？16GB？32GB？你只用了你处理器寻址能力的百分之零点零零零零零零零几。这就像你拥有一整个太平洋，但你只用了一杯水来泡茶。

### 6.2 为什么你的电脑是 aarch64 而不是 x86_64

因为你买了一台 2020 年之后出的 Mac。

在 2020 年之前，Mac 用的是 Intel 处理器，指令集是 x86_64。在 2020 年 11 月，苹果发布了 M1 芯片，指令集变成了 aarch64。这是一次历史性的转变——苹果把 Mac 的"大脑"从一家公司（Intel）换成了另一家——不对，换成了自己（Apple Silicon）。

这就像你开了十五年的本田，突然有一天你把发动机换成了特斯拉的电机——但方向盘还是本田的，座椅还是本田的，后视镜还是本田的，所以你开起来感觉差不多，但每次你踩油门的时候你的后背会被按在椅背上因为加速太快了。

M1 出来以后，所有软件开发者都面临一个选择：你的软件要编译成 x86_64（给老 Mac 用）还是 aarch64（给新 Mac 用）还是 universal（两个都支持但安装包大一倍）？

这个软件的作者选择了只编译 aarch64。为什么？可能是因为：

1. 作者自己用的是 Apple Silicon Mac，所以他本地编译出来的就是 aarch64
2. 作者考虑过做 universal binary，但那意味着要配置交叉编译，而配置交叉编译的时间足够他再写一个新软件了
3. 作者觉得到了 2026 年还在用 Intel Mac 的人应该有能力自己解决问题
4. 作者根本不知道还有 Intel Mac 用户——就像你不知道还有人在用 Windows XP 一样（他们存在的。他们在某个政府部门的地下室里。他们的电脑上运行着一个 2004 年编写的 VB6 程序。那个程序仍然在正常工作。没有人敢碰它。）

### 6.3 aarch64 用户的自我认知

如果你能成功安装并运行这个软件，那你就是一个 aarch64 用户。这意味着什么？

这意味着你的电脑大概率比较新。你大概率花了至少八千块人民币买了这台电脑。你大概率在苹果发布会上看到过 Tim Cook 用一种平静到近乎催眠的语调说"Apple Silicon"这个词。你大概率在第一次启动你的 M1/M2/M3/M4 Mac 的时候惊叹过"这电池怎么这么耐用"。你大概率在某个时刻用你的 Mac 烤过苹果——不是真的烤苹果，而是发现你的 Mac 即使在高负载下也不怎么发热，然后怀念起你之前那台 Intel MacBook Pro 可以在大腿上煎鸡蛋的日子。

你是一个 aarch64 用户。你属于一个精英群体——这个群体的入门门槛是"在 2020 年之后买了一台 Mac"。这听起来不是很精英，但在那些还在用 2015 年款 MacBook Pro 的人眼中，你就是精英。

恭喜你。你的精英身份已经被一个名字很长的免费 Markdown 编辑器官方认证了。

## 第七章：关于 .dmg 这个安装格式

### 7.1 人类安装软件简史

让我们把时间轴拉长一点。很长一点。

**1970 年代**：安装软件意味着把一叠打孔纸带送进读卡机。如果纸带中间折了一下，你的程序就会出现一个莫名其妙的 bug，而你需要花三天时间才能定位到是第 7,342 个孔的位置偏了 0.3 毫米。

**1980 年代**：安装软件意味着把一张 5.25 英寸软盘插进软驱。软驱发出"嗞嗞嗞嗞"的声音。你盯着屏幕上的进度条。进度条走到 87% 的时候停住了。你心跳加速。两秒后它又动了。你松了口气。你的寿命因为这两秒钟缩短了大约三天。

**1990 年代**：安装软件意味着插入一张 CD-ROM，然后点击 `setup.exe`。一个安装向导弹出来。它问你："请选择安装路径。"你看着默认的 `C:\Program Files\AppName`，想了三秒钟，然后点了"下一步"。它又问你："请选择组件。"你看着一堆你不认识的组件名字——"核心文件"、"帮助文档"、"示例项目"、"桌面快捷方式"、"开机自启动"——你全选了。你永远不会打开帮助文档。你永远不会看示例项目。但你选了。因为万一以后需要呢？（你不会需要的。）然后安装向导告诉你需要重启电脑。为什么安装一个软件需要重启电脑？你不知道。Windows 也不知道。但你照做了。因为你没有选择。

**2000 年代**：安装软件意味着从网上下载一个 `.exe` 文件。你的杀毒软件弹出了一个警告。你点了"信任"。你的杀毒软件又弹出了一个警告。你又点了"信任"。你的杀毒软件第三次弹出了一个警告。你把杀毒软件卸载了。然后你发现那个 `.exe` 文件真的是病毒。

**2010 年代**：安装软件意味着打开 App Store，搜索应用名，点击"获取"，等待下载，完成。或者意味着 `brew install xxx`。或者 `apt-get install xxx`。或者 `npm install -g xxx`。或者 `pip install xxx`。或者 `cargo install xxx`。每个生态系统都有自己的包管理器。你的电脑里有至少五个包管理器。它们互相不知道对方的存在。它们各自管理着各自的依赖。你的电脑像一个有五个家长的孩子——每个家长都觉得自己负责你的全部，但实际上没有一个家长知道其他四个给你灌了什么。

**2020 年代**：安装软件意味着双击一个 `.dmg` 文件，拖一下图标。

你看，人类花了五十年时间，从打孔纸带进化到了"拖一下图标"。五十年。两代人。无数工程师的心血。就是为了让你——一个只想打开 `.md` 文件的普通人——能够在三秒钟内完成软件安装。

致敬。

### 7.2 DMG 的仪式感

回到正题。`.dmg`。

DMG 是 Disk Image 的缩写。直译过来是"磁盘镜像"。但你的 Mac 里已经没有磁盘了——它用的是固态硬盘。所以"磁盘镜像"这个名字本身就是一个历史遗留。就像我们还在说"挂电话"（hang up the phone），但你已经十年没有见过那种可以挂起来的电话了。语言比技术进化得慢。我们的词汇还停留在上一个时代，但我们的硬件已经跑到了下一个时代。

当你双击一个 `.dmg` 文件的时候，macOS 会做以下事情：

1. 挂载一个虚拟磁盘（就像插入了一张虚拟的 CD）
2. 弹出一个窗口，里面通常有两样东西：一个 App 图标和一个 Applications 文件夹的替身（alias）
3. 窗口的背景可能是一张精心设计的图片，上面画着一个从左到右的箭头，暗示你把 App 图标拖到 Applications 文件夹上

这个设计是天才级的用户体验。它不需要任何文字说明。一个箭头就够了。从左到右。把这个东西放到那个地方。三岁小孩都能理解。

但你知道最讽刺的是什么吗？大约 37% 的用户（我瞎编的数据但你觉得挺可信）会跳过拖拽步骤，直接双击 `.dmg` 里面的 App 图标来运行软件。然后他们会发现软件是从虚拟磁盘上运行的，速度有点慢，而且每次重启电脑后都需要重新挂载 `.dmg`。于是他们会在两周后发一条朋友圈说"这个软件怎么这么难用"。

不。不难用。你只是没有把它拖到 Applications 里。

### 7.3 关于 Gatekeeper 的灵魂拷问

当你第一次打开这个软件的时候，macOS 会弹出一个对话框。对话框上面写着：

> "「因为产出了很多MD文档所以需要一个简单免费的MD文件查看编辑软件」无法打开，因为它来自身份不明的开发者。"

让我们逐词分析这句话的潜台词：

- **"无法打开"**：不是真的"无法"。是"不愿意"。macOS 完全有能力打开它，但它选择了不。就像你妈问你"你能不能少打点游戏？"——她知道你能，她只是不让你。
- **"因为它来自身份不明的开发者"**：什么叫"身份不明"？作者有名有姓。作者有 GitHub 账号。作者可能还有 LinkedIn。他不是一个匿名黑客。他只是没有花 $99/年 注册一个 Apple Developer 账号。在苹果的世界观里，一个不愿意每年花 $99 的人就是"身份不明"。

这个对话框本质上是苹果在说："你确定你要信任一个连 99 美元都不愿意花的开发者吗？"

你确定。

你去「系统设置 → 隐私与安全性」里找到那行小字，点击"仍然打开"。macOS 又问了一遍："你确定吗？"你说确定。macOS 沉默了一秒——也许它在失望，也许它在叹气——然后它放行了。

软件打开了。

一切正常。世界没有毁灭。你的电脑没有爆炸。你的银行卡没有被盗刷。那个"身份不明的开发者"只是做了一个 Markdown 编辑器而已。

但 Gatekeeper 不会道歉。下次你安装另一个未签名的软件，它还会弹出同样的对话框。它永远不会学到"这个用户知道自己在做什么"。它就像你公司的安保，每天都要你刷工牌进门——即使你已经在这里工作了八年，安保大叔每天都能看到你的脸，他还是会说："请出示你的工牌。"

这就是安全。安全不讲情面。安全不记人脸。安全只认工牌。或者在 macOS 的语境下——安全只认 $99。

## 第八章：关于这个名字

### 8.1 命名学导论

在人类文明的长河中，命名一直是一件严肃的事情。

给孩子取名，父母要翻字典、查典故、算生辰八字、请大师测吉凶。给公司取名，创始人要注册商标、测试发音、考虑国际化、避免在其他语言里的不雅含义（比如某个汽车品牌在西班牙语里的意思是"走不动"，另一个在粤语里的意思就不说了）。给宠物取名——好吧，给宠物取名大多数人花了三秒钟。"就叫它'咪咪'吧。""好。"

给软件取名呢？这介于给孩子取名和给宠物取名之间。你希望它有意义，但你也知道没有人真的在意你的软件叫什么——他们只在意它能不能解决问题。

历史上那些伟大的软件名字是怎么来的？
